[[external-properties]]
= External Properties

ArcadeDB stores all of a record's properties inline by default: when a vertex, edge or document is loaded from disk, the entire payload is materialised in memory.
This is the right trade-off for small records, but it becomes expensive when one or two properties dominate the record size, for example a 4096-element float embedding (~16 KB per record), a multi-kilobyte JSON document, or a long text body.
A traversal that only walks the graph or filters on a small property still pays the I/O and deserialisation cost of those heavy fields.

`EXTERNAL` properties solve this problem.
When a property is marked as `EXTERNAL`, its value is stored in a separate "paired" bucket keyed by the same record's RID, and the primary record only carries a small `TYPE_EXTERNAL` pointer.
Reads that do not project the property never touch the paired bucket; reads that do project it perform one extra page fetch to materialise the value.

== When to use

Use `EXTERNAL` when *all* of the following are true:

* The property's value is large compared to the rest of the record (typically more than a few hundred bytes).
* The property is read selectively, not on every query against the type.
* You are willing to pay an extra I/O when the property *is* projected, in exchange for cheaper traversals when it is not.

Typical good fits:

* Vector embeddings (`ARRAY_OF_FLOATS`).
* Long-form text fields (article bodies, descriptions, blob payloads).
* Embedded JSON documents or maps.
* Full-text indexed payloads.

Avoid `EXTERNAL` for small, frequently-projected properties: the extra page fetch will outweigh any savings.

== Storage layout

For each primary bucket of a type that owns at least one `EXTERNAL` property, ArcadeDB creates a paired *external bucket*:

* Named after the primary bucket with the suffix `_ext` (for example, `Doc_0_ext` pairs with `Doc_0`).
* Marked with the purpose `EXTERNAL_PROPERTY` and excluded from the type's regular bucket list, so writes via `INSERT INTO bucket:<name>` cannot target it directly.
* Sized with a 256 KB page (vs. 64 KB for primary buckets) and a smaller slot table (256 vs. 2048 records per page) so multi-KB payloads fit in a single page rather than spilling into the multi-page chunk chain. The page size matches the LSM-tree default.
* The paired buckets are visible in the `schema:buckets` system view via the `purpose` column.

Inheritance is handled transparently: when a subtype is added to a parent that owns an `EXTERNAL` property, ArcadeDB ensures every primary bucket of the subtype also has its own paired external bucket.

[NOTE]
====
The xref:concepts/transactions.adoc[Write-Ahead Log] covers both halves of a write together: a rolled-back transaction reverts the primary record *and* the paired external blob.
A delete cascades to the paired blob, so there is no manual cleanup to do under normal operation.
====

== Compression

Each `EXTERNAL` property has an independent compression policy:

[cols="1,3",options="header"]
|===
| *Policy* | *Description*
| `none` (default) | Store the value raw.
| `fast` | LZ4 fast encoder. The right default when writes are frequent. Decompression speed is the same as `max`.
| `max` | LZ4 HC (High Compression) encoder. Roughly 10 percentage points smaller than `fast`, 8 to 20x slower to compress, identical to decompress. Best for write-once / read-many payloads where bucket file size dominates.
| `auto` | Try `fast`; keep the compressed bytes only when they save more than 10% of the raw size, otherwise fall back to raw. Useful for mixed workloads where some records compress and others (e.g. dense float32 embeddings) do not. Pays one wasted compress + one extra byte-array copy on no-win records.
|===

The legacy alias `lz4` is accepted on input and stored as `fast`.

== Tiered storage

By default, the paired external buckets live alongside the primary buckets in the database directory.
You can relocate them to a different filesystem (cheaper or slower storage, network mount, etc.) per database via the xref:reference/settings.adoc#settings-sql[`arcadedb.externalPropertyBucketPath`] setting.
The directory must exist and be writable; existing external buckets are not moved when this setting changes, so set it before creating the type or rebuild the type after moving the files.

== Toggling `EXTERNAL` on existing data

The `EXTERNAL` flag affects only the *write* path: existing records keep their old layout until they are re-saved.
Reads continue to work because the deserialiser detects the inline layout from the bytes.

To migrate existing records to the new layout, run xref:reference/sql/sql-rebuild-type.adoc#sql-rebuild-type[`REBUILD TYPE`]:

[source,sql]
----
ALTER PROPERTY Doc.embedding EXTERNAL true;
REBUILD TYPE Doc;
----

When a property is switched back from `EXTERNAL true` to `EXTERNAL false`, `REBUILD TYPE` re-serialises the records inline and drops the now-empty paired buckets so they do not accumulate across toggle cycles.

== Operational notes

* *Crash recovery:* a crash between writing the external value and re-serialising the primary record can leave orphaned blobs in the paired bucket. Run xref:reference/sql/sql-check-database.adoc#sql-check-database[`CHECK DATABASE FIX`] to detect and remove them. The `CHECK DATABASE` report exposes the counters `orphanedExternalRecords` and `orphanedExternalRecordsFixed`.
* *Indexes:* an index on an `EXTERNAL` property works exactly as on an inline property; the index entries are extracted at write time and stored normally.
* *Null and remove:* `set("field", null)` on an `EXTERNAL` property does not consume external bucket space. Any pre-existing paired blob is released through the orphan-cleanup pass. `set(field, null)` and `remove(field)` are equivalent in terms of bucket storage; the difference is whether the property header slot is retained.
* *Direct writes are forbidden:* the paired external buckets reject `INSERT INTO bucket:<name>` and the Java `createRecord(record, bucketName)` path. The only supported write path is through the owning record's serialisation.

== SQL examples

[source,sql]
----
-- Create a type with an EXTERNAL vector embedding (raw, uncompressed)
CREATE DOCUMENT TYPE Doc;
CREATE PROPERTY Doc.name STRING;
CREATE PROPERTY Doc.embedding ARRAY_OF_FLOATS (EXTERNAL true);

-- Create an EXTERNAL text property with auto-compression
CREATE PROPERTY Doc.body STRING (EXTERNAL true, COMPRESSION 'auto');

-- Promote an existing inline property to EXTERNAL and migrate the records
ALTER PROPERTY Doc.body EXTERNAL true;
REBUILD TYPE Doc;

-- Switch the compression policy
ALTER PROPERTY Doc.body COMPRESSION 'max';

-- Roll back: bring the property inline and reclaim the paired buckets
ALTER PROPERTY Doc.body EXTERNAL false;
REBUILD TYPE Doc;
----

== Java API

`EXTERNAL` and `compression` are properties of the `Property` interface and behave like the other constraints (`mandatory`, `notnull`, etc.):

[source,java]
----
final DocumentType type = database.getSchema().createDocumentType("Doc");
type.createProperty("name", Type.STRING);
type.createProperty("embedding", Type.ARRAY_OF_FLOATS).setExternal(true);
type.createProperty("body", Type.STRING).setExternal(true).setCompression("auto");

// Toggle off and reclaim
type.getProperty("body").setExternal(false);
database.command("sql", "REBUILD TYPE Doc");
----
