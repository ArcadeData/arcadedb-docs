[[sql-rebuild-type]]
[discrete]
= SQL - `REBUILD TYPE`

Re-serialises every record of a type so that schema changes that affect the on-disk binary layout are applied to records created before the change.
The most common use case is toggling the xref:concepts/external-properties.adoc#external-properties[`EXTERNAL`] flag on a property: existing records keep the old layout until they are re-saved, and `REBUILD TYPE` performs that re-save in batches.

*Syntax*

[source,sql]
----
REBUILD TYPE <type-name> [POLYMORPHIC] [WITH batchSize = <N>]
----

* *`<type-name>`* The type whose records must be re-serialised.
* *`POLYMORPHIC`* If specified, the rebuild walks the type and all of its subtypes.
* *`batchSize`* Number of records committed per intermediate transaction.
The default is `10000`.
Larger batches are faster but hold more memory and produce a larger Raft log entry per commit when running under HA.
The value must be a positive integer.

The command returns one record with the following fields:

[cols="1,3",options="header"]
|===
| *Field* | *Description*
| `operation` | Always `rebuild type`.
| `typeName` | Name of the type being rebuilt.
| `polymorphic` | `true` when the rebuild walked the subtypes too.
| `recordsRebuilt` | Total number of records re-serialised.
| `warning` | Optional message present only when paired external buckets could not be reclaimed because the command was issued inside a caller-supplied transaction. Re-run `REBUILD TYPE` outside any transaction to drop the now-empty buckets.
|===

*Behaviour*

* When called outside of a transaction (the default in the console), `REBUILD TYPE` manages its own transactions and commits every `batchSize` records. This means the command is *not* atomic across batch boundaries: a failure mid-rebuild leaves the type half-migrated. The error message reports how many records were already committed in the new layout and how many were rolled back. Re-run `REBUILD TYPE` once the underlying problem is fixed to migrate the remaining records.
* If a property has just been switched from `EXTERNAL true` back to `EXTERNAL false`, `REBUILD TYPE` also drops the now-empty paired external buckets so that toggling does not accumulate unused buckets across cycles.
* If a rebuild fails in a way that left orphaned blobs in the paired external bucket (for example, the process crashed between writing the external value and re-serialising the primary record), run xref:reference/sql/sql-check-database.adoc#sql-check-database[`CHECK DATABASE FIX`] to delete them.
* Under Raft High Availability, `REBUILD TYPE` is forwarded to the leader and replicated to followers as a sequence of standard transactions, one per batch.

*Examples*

* Re-serialise every record of type `Doc` after toggling a property's `EXTERNAL` flag:

[source,sql]
----
ArcadeDB> ALTER PROPERTY Doc.embedding EXTERNAL true
ArcadeDB> REBUILD TYPE Doc
----

* Rebuild a parent type and all of its subtypes:

[source,sql]
----
ArcadeDB> REBUILD TYPE Person POLYMORPHIC
----

* Rebuild with a smaller batch size to reduce per-commit memory pressure:

[source,sql]
----
ArcadeDB> REBUILD TYPE Doc WITH batchSize = 1000
----
