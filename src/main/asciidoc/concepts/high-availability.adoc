=== High Availability

ArcadeDB is designed for continuous operation.
Its high availability (HA) architecture ensures that your database remains accessible even when individual servers fail, providing both fault tolerance and horizontal read scalability.

ArcadeDB implements HA using the **Apache Ratis** library, which provides a production-grade implementation of the **Raft consensus protocol**.
This ensures that your cluster maintains data consistency and availability even in the face of network partitions or node failures.

This page explains the concepts behind ArcadeDB's HA architecture.
For step-by-step setup instructions, see <<howto-high-availability,HA Configuration>>.

==== Leader-Replica Replication Model

ArcadeDB uses a leader-replica replication model.
At any point in time, a cluster has exactly one *leader* server and one or more *follower* (replica) servers.

* The **leader** coordinates all write operations and distributes changes to followers.
* **Followers** maintain a copy of the data and can serve read requests independently, allowing the cluster to scale read throughput horizontally.

Clients can connect to any server in the cluster.
If a client sends a write request to a follower, the follower transparently forwards it to the leader.
This means applications do not need to distinguish between leader and follower servers -- the cluster handles routing internally.

==== Raft Consensus and Leader Election

ArcadeDB uses the Raft protocol to manage the cluster state and ensure consensus.

* **Leader Election**: When a cluster starts, or if the current leader fails, the nodes initiate an election. Raft guarantees that only one leader is elected at a time and that the new leader has the most up-to-date log.
* **Log Replication**: Every write operation is recorded as a log entry in the Raft log. The leader replicates these log entries to all followers. Once a majority of nodes have acknowledged the entry, it is considered committed.

If a server fails, it can rejoin the cluster later. Raft automatically detects the new server and synchronizes its state with the leader, either by replaying missing log entries or by installing a fresh database snapshot.

==== Physical Replication (WAL-based)

ArcadeDB uses **physical replication** to ensure that all nodes in the cluster have an identical bit-for-bit copy of the database.

When the leader processes a write operation, it serializes the **Write-Ahead Log (WAL)** page diffs and sends them to the followers via Raft.
The followers apply these physical page changes to their local storage, ensuring that the database state is exactly the same across the entire cluster.

This mechanism is extremely fast and efficient, as it bypasses high-level logic and operates directly at the storage layer.

==== Write Quorum and Consistency

ArcadeDB uses a configurable *write quorum* to control the trade-off between consistency and availability.
The quorum defines how many servers must acknowledge a write before it is considered committed.

The default quorum is `MAJORITY`, meaning more than half of the servers in the cluster must confirm the write.
This default provides a strong balance: it ensures data durability across multiple servers while tolerating the failure of a minority of nodes.

Other quorum options include:

* `none` -- the leader commits immediately without waiting for any follower (fastest, least durable).
* `all` -- waits for every server in the cluster to acknowledge (strongest durability, highest latency).

For more on how transactions interact with replication, see <<transactions,Transactions>>.
For quorum and other cluster settings, see <<arcadedb-settings,Server Settings>>.

==== Automatic Failover

ArcadeDB handles leader failure automatically.
When followers detect that the leader is no longer reachable (via Raft heartbeats), they initiate a new election.
The node with the most up-to-date log is elected as the new leader, and the cluster resumes normal operation.

This failover process is transparent to clients.
After a brief interruption during the election, clients reconnect to the new leader (or are forwarded by a follower) and continue operating without manual intervention.

==== Snapshot-based Resync

If a follower falls too far behind the leader (e.g., it was offline for a long period and the required Raft log entries have been compacted), ArcadeDB uses **snapshot-based resync**.

The leader creates a consistent snapshot of the database and transmits it to the follower over HTTP.
The follower installs this snapshot, effectively resetting its database to the state of the leader, and then continues to synchronize using the Raft log.

==== Split-Brain Protection

A *split-brain* occurs when a network partition divides a cluster into two or more groups, each believing it should operate independently.
This can lead to conflicting writes and data divergence.

ArcadeDB prevents split-brain scenarios through the Raft protocol's majority requirement.
Because a leader must be elected by a majority and must receive acknowledgment from a majority to commit writes, at most one partition can ever be active for writes.
Any minority partition cannot elect a leader or commit writes, so it effectively becomes read-only (or waits for reconnection) until the network heals.

This design guarantees that the cluster never produces conflicting committed data, even during network failures.
