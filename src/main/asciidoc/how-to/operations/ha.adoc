==== High Availability
image:../../images/edit.png[link="https://github.com/ArcadeData/arcadedb-docs/blob/main/src/main/asciidoc/how-to/operations/ha.adoc" float="right"]

ArcadeDB supports a High Availability mode where multiple servers share the same database via replication using the **Apache Ratis** implementation of the **Raft consensus protocol**.

NOTE: All servers of a cluster serve the same databases.

In order to start an ArcadeDB server with High Availability enabled, you need to:

1. Enable the HA module by setting the configuration `arcadedb.ha.enabled` to `true`.
2. Define the servers that are part of the cluster.
To set up the server list, define the `arcadedb.ha.serverList` setting as a comma-separated list of servers in the following format: `<hostname/ip-address[:raftPort][:httpPort][:priority]>`.
    * **raftPort** is the port for Raft communication (default is `2434`).
    * **httpPort** is the port for HTTP communication, which is required for command forwarding (default is `2480`).
3. Define the local server name by setting the `arcadedb.server.name` configuration. Each node must have a unique name. If not specified, the default server name is `ArcadeDB_0`.
4. (Recommended) Set a shared **cluster token** by setting `arcadedb.ha.clusterToken`. This is a secret shared between all nodes for authentication of forwarded commands.

Example:

[source,shell]
----
$ bin/server.sh -Darcadedb.ha.enabled=true \
                -Darcadedb.server.name=FloridaServer \
                -Darcadedb.ha.serverList=node1:2434:2480,node2:2434:2480,node3:2434:2480 \
                -Darcadedb.ha.clusterToken=mysecrettoken
----

The log should look like this:

[source,shell]
----
[HAServer] <FloridaServer> ArcadeDB HA (RAFT) stack starting...
[HAServer] <FloridaServer> Configuration:
- Cluster Name: arcadedb
- Server List: node1:2434:2480,node2:2434:2480,node3:2434:2480
- Quorum: MAJORITY
- Local Raft Port: 2434
[HAServer] <FloridaServer> Starting Raft server on 0.0.0.0:2434 (Ratis v.3.2.2)
[HAServer] <FloridaServer> Attempting to join cluster...
[HAServer] <FloridaServer> New leader elected: node1
[HAServer] <FloridaServer> Cluster joined successfully. Current role: FOLLOWER
----

At startup, the ArcadeDB server will look for an existing cluster to join based on the configured list of servers. If no cluster exists, a new cluster will be created and an election will take place.

Every time a server leaves the cluster (becomes unreachable), the Raft election process ensures that a new leader is elected automatically.

NOTE: The cluster name defaults to "arcadedb". You can specify a custom name via `arcadedb.ha.clusterName`.

Each server maintains its own Raft log, which records all modifications. The leader replicates these entries to followers.

Any read operation, like a query, can be executed by any server in the cluster. Write operations are always coordinated by the leader.

===== Write Quorum

The write quorum controls how many nodes must acknowledge a write before it is committed.

* `MAJORITY` (default): More than half of the servers must acknowledge. Provides a balance between durability and availability.
* `NONE`: The leader commits immediately. Highly performant but lacks durability guarantees on followers.
* `ALL`: Every server in the cluster must acknowledge. Strongest durability but highest latency.

Example to set the quorum:
[source,shell]
----
$ bin/server.sh -Darcadedb.ha.enabled=true -Darcadedb.ha.quorum=all
----

===== Monitoring Replication Lag

You can monitor the replication lag of followers to ensure the cluster is performing optimally.
The leader tracks the difference between its current log index and the last index successfully applied on each follower.

If a follower falls behind by more than `arcadedb.ha.replicationLagWarning` entries (default 1000), a warning is logged by the leader.

You can also check the cluster status via the HTTP API:

[source,shell]
----
$ curl -u root:playwithdata http://localhost:2480/api/v1/cluster
----

The output includes information about each node, their current role, and their replication lag.

===== Automatic Failover

If the leader node becomes unreachable, the followers will automatically detect the failure and elect a new leader using the Raft protocol. This process typically takes a few seconds.

During an election, write operations will fail or be queued until a new leader is established. Read operations from existing followers are still available, though they may serve slightly stale data depending on the configured **read consistency**.

===== HA Settings

The following settings are used by the High Availability module:

[%header,cols=3]
|===
|Setting|Description|Default Value
|`+arcadedb.ha.enabled+`|True if HA is enabled for the current server|false
|`+arcadedb.ha.clusterName+`|Cluster name.|arcadedb
|`+arcadedb.ha.serverList+`|List of <hostname/ip-address:raftPort:httpPort[:priority]> separated by comma.|NOT DEFINED
|`+arcadedb.ha.raftPort+`|Default TCP/IP port for Raft communication.|2434
|`+arcadedb.ha.clusterToken+`|Shared secret for inter-node command authentication.|NOT DEFINED
|`+arcadedb.ha.quorum+`|Default quorum: `none`, `majority`, `all`.|majority
|`+arcadedb.ha.quorumTimeout+`|Timeout waiting for the quorum in milliseconds.|10000
|`+arcadedb.ha.readConsistency+`|Read consistency for follower reads: `eventual`, `read_your_writes`, `linearizable`.|read_your_writes
|`+arcadedb.ha.replicationLagWarning+`|Log warning if replication lag exceeds this many entries.|1000
|`+arcadedb.ha.raftSnapshotThreshold+`|Number of entries after which a snapshot is automatically taken.|10000
|`+arcadedb.ha.logVerbose+`|HA logging level: 0=off, 1=basic, 2=detailed, 3=trace.|0
|`+arcadedb.ha.k8s+`|The server is running inside Kubernetes.|false
|`+arcadedb.ha.k8sSuffix+`|Kubernetes DNS suffix for reaching other servers.|
|===
